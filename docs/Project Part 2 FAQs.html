General Question

  1. There is a page limit for the reports. What if I exceeded the page limit?
    • The teaching team will only mark the pages within the page limit, subsequent pages will be ignored.
  2. I’m not sure what additional features I can implement for the project. What should I do?
    • Please refer to the suggestions provided in the project specifications. For each part of the project, there is a guided additional feature which you can try. You can them in the  Additional Features Guide.

Project Part 1

  1. Is this an individual or group project?
    • Due to the nature of the subject, you would need to form a group to work on Project Part 1.
    • However, each student is assessed individually. You will implement the code for your own proposed feature  independently and write your own report.
  2. What is considered a "feature" in the context of this project?
    • A feature refers to any CRUD operation (Create, Read, Update, Delete).
    • The feature must demonstrate a complexity level ranging from average to high. 
  3. How is the complexity level of average to high defined?
    • As a guide, features that interact with the database and encompasses alternate scenarios (success and at least two other error cases) will typically meet the required complexity level. 
    • E.g. Create resource is a suitable function as the function can
      • contain multiple alternate scenarios, such as:
        • All required fields properly filled up
        • Some required fields not filled up
        • All required fields filled up but contains invalid data (e.g. NRIC number is invalid, etc) 
      • and interacts with the database as the data will be inserted.
    • E.g. A typical "delete resource" on the other hand is not a good function as it doesn't encompass much alternate scenario. However, it will fulfill the complexity level if it's enhanced with a confirmation field before deletion.
  4. If my groupmate had already chosen a "Create" feature, can I implement another "Create" feature.
    • Yes, you may, as long as it is not the same feature.
    • E.g. Your groupmate can work on "Create blog post" while you work on "Create comment".
  5. Can I implement more than one feature?
    • No, each student is only required to implement one feature. There are no extra marks for implementing additional features. In the event that you chose to go against this and implement more than one feature, the teaching team will only mark the first feature that we see in your report (which might not be your best feature).
    • However, if supporting features are needed, you may implement them but you should state clearly in your report on the chosen feature that the teaching team will mark.
  6. What are the deliverables for Project Part 1?
    • Each student is required to submit a Word document report.
    • Each group is required to submit their source code, PPT slides and do a presentation.
  7. What should we do if one teammate is not cooperating or hasn't completed their part of the project on time, even though we're working collaboratively on GitHub and had set realistic milestones and deadlines?
    • This is why clear communication, regular updates and a shared project plan are essential in collaborative work. If a teammate is falling behind beyond the agreed timeline and there's evidence of ongoing, documented communication, the rest of the team can proceed:
      • Go ahead and merge the completed features into the main branch for submission.
      • Note the commit ID of the final working version.
      • Submit that version as the group’s main contribution.
    • The teammate who is behind can complete their part and merge their work separately later, creating a second copy of the code for submission if necessary.
    • Please also inform your tutor so that he/she can speak to your teammate and expect multiple code submissions from your team.

Project Part 2

  1. Do I need to continue collaborating with my groupmate for Project Part 2?
    • No, you do not need to. 
    • You will fork the repository from Part 1 and work independently from Project Part 2 onwards.
      • Forking a repository means you can make any changes you want to the code without affecting the original project. 
  2. How do I fork the repository from Part 1 into my own repository?
    • Please refer to the video recording in the Project Part 2 briefing announcement.
  3. Can I modify my groupmates' features after forking the repository?
    • Yes, you can make changes to your team members' features, but you will not receive extra marks for this.
  4. Can I modify my own feature after forking the repository for Project Part 2?
    • Yes, you can modify your own feature to for example, cover more alternative scenarios, but you are not allowed to change the core functionality of the function.
  5. What testing is required for Project Part 2?
    • You must perform backend unit testing, API testing and frontend testing on your proposed feature and gather the both the backend and frontend code coverage.
  6. Do I need to perform testing for my groupmates' features?
    • No, you do not need to.
    • Marks may also be deducted due to unrepresentative code coverage.
  7. Do I need to gather overall code coverage for the entire application?
    • No, you do not need to as you are only expected to perform testing on your proposed feature.
  8. Can I use testing frameworks other than Jest and Playwright?
    • You may propose and implement another backend or front-end testing framework, but you must also implement backend unit testing and API testing using Jest and front-end testing using Playwright.
  9. What are the deliverables for Project Part 2?
    • Each student is required to submit a Word document report and their source code.
    • You are also required to do a demonstration of your testing and code coverage implementation.
  10. For the additional feature, if I implement an alternative testing tool or framework, am I required to rewrite all the existing test cases that were originally written using Jest/Playwright?
    • Yes. All existing test cases should be rewritten in the new testing tool or framework to ensure completeness.
  11. There are multiple code coverage metrics. Which ones will I be evaluated on? 
    • All of them. Ensure that every coverage metric meets or exceeds the target percentage you are aiming for.

Project Part 3

  1. What is required for Project Part 3?
    • You are required to implement CI/CD pipeline to automate build, test and deployment.
  2. Is it okay to use tools other than Jenkins, Docker, and Minikube?
    • You may propose and implement another CI/CD, containerization, and container orchestration tool, but you must also implement the CI/CD pipeline using Jenkins, Docker and Minikube.
  3. What are the deliverables for Project Part 3?
    • Each student is required to submit a Word document report and their source code.
    • You are also required to do a demonstration of your CI/CD pipeline implementation.
  4. What if I couldn't manage to implement any additional feature?
    • You can comment out the code (if it's breaking your application or pipeline) and still write about it in under the "Additional features" section of your report and document your planning, research, implementation and reflection process in the "PPMR" section of the report.
